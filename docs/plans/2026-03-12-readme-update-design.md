# README.md 更新デザインドキュメント

## 概要

最近追加された「内部ツールの露出制御・自動承認」および「動的モデル対応機能」を `README.md` に反映し、ユーザーが新機能の存在と背後の挙動を正しく理解できるようにする。

## 目的と制約

- **目的**: A2A内部ツール（`activate_skill`等）の自動承認などの高度な制御が実装されたこと、および動的モデルのサポート状況をドキュメント化する。
- **制約**: 現在のREADMEの構成を大きく崩さないこと。「特徴」リストに簡潔に追記し、詳細は新設する「高度な設定・機能」セクションで説明する。

## 更新内容の設計

### 1. 「## 特徴」セクションへの追記
以下の項目をリストに追加する。
- **内部ツールの自動承認とUI非表示制御**: A2Aの内部ツール（`activate_skill`等）をOpenCodeのUIに露出させず、プロバイダー層で自動承認ループを回すことで、シームレスな対話とツール実行の中断エラーを防ぎます。
- **動的モデルのサポート**: 最新のモデルリストを動的に解決・ルーティングする機能を提供します。

### 2. 「## 高度な設定・機能」セクションの新設
「## 使い方」または「## 制限事項および残存課題」の前後（構成上自然な位置）に新設し、以下の詳細を記述する。
- **内部ツールの自動承認ループ**:
  A2Aサーバーが要求する `tool-call-confirmation` などの内部状態に対し、プロバイダーは最大10回のループを回して自動的に確認リクエスト（`buildConfirmationRequest`）を送信します。これにより、OpenCodeのUIが不要なツールコールを受け取ってクラッシュすることを防いでいます。
- **デバッグログの出力制御**:
  環境変数 `DEBUG_OPENCODE=1` などを設定することで、リクエスト/レスポンスの詳細やツールコールの自動承認カウントなどをコンソール（stderr等）に出力でき、動作のトラブルシューティングに役立ちます。
